🚨 Critical Risks

Risks That Could Invalidate the Architecture

Git commit failures are silent unless explicitly checked If the Node process lacks write permission or git is absent from PATH, all version history silently disappears. Every git call must check exit code and surface a UI toast. File is written first — history is the secondary concern.
Claude may return malformed JSON or partial responses Claude is an external dependency. A timeout mid-stream, a schema deviation, or a rate limit error can corrupt the extraction pipeline. Every response must be validated against the JSON schema before any file is touched. On failure: mark extraction as failed in SQLite, show retry UI.
Concurrent edit conflict: rep edits a field while an AI proposed edit for the same file is pending If the rep's direct-edit save races with an Accept of a proposed edit, one write will silently overwrite the other. Mitigation: backend checks SQLite for pending_direct_edit rows before accepting; returns 409 with a confirmation modal.
Frontend holds stale file content during direct edits The debounce save sends the full file content from the frontend's in-memory copy. If Claude accepted an edit to the same file while the rep had it open, the rep's save will overwrite Claude's changes. Fix: backend must re-read current file and merge, or frontend must re-fetch before saving.
Mockup migration regressions The existing mockup has 100% of the desired UX already built. Adding fetch() calls risks breaking tab state, panel layouts, and diff rendering. Mitigation: Phase 1 preserves all HTML/CSS exactly — JS additions only, no structural rewrites.
The Problem

What We're Building and Why

Sales reps at Golden Eagle log home company spend 30–90 minute calls with clients learning critical details — floor plan preferences, who the decision-makers are, what the blockers are — and then lose most of it to incomplete notes or memory.

The core problem: Project knowledge lives in the rep's head, not in a system. Every call should update the project record. Almost none of them do, because updating is manual and friction-heavy.

What the CRM does

Post-call extraction Upload call recording → AI reads transcript + screenshots + project files → proposes targeted edits → rep reviews and accepts in one click
Living project documents Floor plan, people profiles, project overview — markdown files that stay current automatically. Human-readable. Directly editable.
Version history Every accepted edit = a git commit. Google Docs-style version panel with inline diffs. Nothing is ever lost.
AI assistant chat "Update Lauren's notes to say she prefers cedar" → AI reads the file → proposes the edit → same review UI → accepted and committed.
Key constraint: AI cannot add document structure (sections, tables). Only content within existing structure. Structure is human-added only. This keeps the rep in control of the document shape.
Architecture

Components and Data Flow

The stack

Frontend: evolved mockup HTML/JS The existing mockup becomes the real app. All CSS and layout preserved. fetch() calls replace hardcoded data. No React, no bundler, no build step.
Backend: Express/Node REST API. Handles file reads/writes, git operations, SQLite queries, Claude API proxy. Runs locally.
Project documents: markdown files floor-plan.md, project-overview.md, people/lauren-thompson.md. Human-readable. AI edits these like source code.
Version history: git Project doc directory is a git repo. Every accepted edit = one commit with a labeled message. git log = version list. git diff = inline diff.
Structured metadata: SQLite Calls table, proposed_edits table (with status), tasks table, events audit log. No document content here — that's in files.

Key Decision 1

Framework: Evolve Mockup vs. Next.js / React

Criterion Evolve Mockup (Express + vanilla JS) ✅ Next.js / React
Preserves existing mockup UI Yes — all HTML/CSS unchanged, add fetch() only No — must port to JSX, risks regressions
Setup time to first working API call ~2 hours ~1 day (scaffolding, routing, components)
Rep can open locally without a build step Yes — just serve the HTML file No — requires npm run build
AI agent can edit without breaking state Yes — simple JS functions, easy to reason about Risky — React state, hooks, component tree
Debuggability Browser console + network tab, direct Same, but compiled output adds a layer
Who owns the fix path if it breaks We own it entirely — no framework layer We own it + framework version issues
Long-term scale (multi-rep) Needs componentization later, but premature now Better for scale — but premature for Phase 1
Decision: Evolve the mockup. The mockup already has 100% of the desired UX. Rebuilding in React destroys that work and adds complexity with zero benefit at current scale.
Key Decision 2

Version History: Git Commits vs. Snapshot JSON Array

Criterion Git commits ✅ Snapshot JSON array in file
Diff computation git diff — exact, line-level, free Must implement custom diff algorithm
Tooling for manual inspection git log, git show, git diff — any developer Requires custom tooling to read JSON array
Human can edit files directly? Yes — git tracks the change Yes, but silently breaks the JSON structure
Corruption failure mode One version lost — rest of history intact JSON parse error = entire history unreadable
Fix path on failure git fsck, git reflog — standard tools Custom code to repair JSON array
Storage growth Text compresses extremely well in git pack files JSON bloat grows unboundedly in the file
Restore semantics Write old content as new commit (append-only) Same append behavior required
Decision: Git commits. The diff tooling is free, the storage format is battle-tested, and the failure mode (one history gap) is far less severe than JSON array corruption making all history unreadable.
Key Decision 3

Storage: Files Only vs. Files + SQLite vs. Database Only

Criterion Files + git only Files + git + SQLite ✅ Database only (Postgres)
Document content In files In files In DB records
Call metadata, proposed edit states, tasks No structured storage SQLite tables DB tables
"Show all pending edits" query Must scan JSON files — not practical Single SQL query Single SQL query
Setup complexity Zero Low — SQLite ships with Node via better-sqlite3 High — DB server, connection string, migrations
DB unavailable failure mode N/A App falls back to file-only reads App is completely down
Right tool for current scale Too limited for real use Yes — fine for 5–10 reps, 50 projects Premature — overkill for current scale
Decision: Files + git + SQLite. Files and git excel at document content and version history. SQLite handles structured metadata without requiring a server. No over-engineering for current scale.

14 Flows — The Critical Branches

The happy paths

Post-call extraction (full pipeline) MP4 + transcript → ffmpeg → Claude Vision → consolidated.txt → Claude extraction → proposed-edits.json → UI module
Transcript-only extraction No MP4 — skip ffmpeg, proceed with transcript only. Fewer context signals but fully functional.
Accept all / Accept single / Skip Per-file or bulk accept. Each accepted file = one git commit with call label. Skip = SQLite update only, file unchanged.
AI chat edit Rep types instruction → same extraction pipeline → same Proposed Edits module → same accept flow.
Direct field edit with debounce 2s debounce → file write → git commit "Rep edit · date". No confirmation needed — version history is the safety net.
Version restore Old content written as a new commit — always append, never overwrite.

The critical error branches

Claude returns invalid JSON Schema validation rejects it. SQLite marked "extraction_failed". UI shows error + retry button. No partial edits written.
Git commit fails File is already written (intentional priority). Log to SQLite events. Show toast. Rep can manually fix git. Content is never lost.
Concurrent edit conflict Backend checks for pending_direct_edit in SQLite before accepting proposed edit. Returns 409. Modal asks rep to confirm.
Claude proposes edit to non-existent file path All file_path values validated against project directory before any write. Path traversal attempts (../../server.js) rejected.
Server restart with pending edits proposed-edits.json never deleted until all rows resolved. UI re-renders from file + SQLite on load. No orphaned pending rows.
Call with no changes detected Claude returns empty array []. UI shows "No proposed edits — no changes detected." No confusion, no module shown.

Engineering Review

Principles Checklist Results

State & Data Integrity

✅ Every state has an exit path Proposed edits: pending → accepted/skipped/failed. Calls: uploaded → processing → ready/failed. No limbo states. Timeout enforced on extraction.
✅ Core logic in durable layers Document content in files + git. Proposed edit state in SQLite. Nothing critical lives only in application memory.

Ownership & Dependencies

✅ We own every trigger No third-party webhooks in the critical path. Rep action → our endpoint. No black boxes in the write path.
✅ Not reimplementing platform features Using git for versioning, SQLite for persistence, ffmpeg for video processing. Not reinventing any of these.

Execution Model

✅ No daemons where cron/event would suffice Express handles requests synchronously. ffmpeg is a child process that exits. No while-True loops. No scheduling.

Failure & Recovery

✅ Retry logic for external calls Claude API: retry × 3 with backoff. File writes: synchronous, no retry needed. Git failure: file already written, log and notify.
✅ Audit trail is downstream SQLite events table written AFTER git commit succeeds. Not used as a trigger. Receipt, not upstream controller.

Observability

✅ Failures surface to the rep immediately Single-user local tool. Errors appear as toasts and error banners, not in server logs the rep never sees.
⚠️ Catch-all monitors deferred Accepted risk for Phase 1. SQLite sanity query available for manual use. Will add background monitoring in Phase 3 if background jobs are introduced.

Agent Safety

✅ Core logic in durable layers An AI agent editing the Express server code cannot corrupt project documents or version history. CLAUDE.md will point to this architecture doc.
Blind Spot Review

8 Findings from Fresh-Perspective Review

Prompt: "You are a different engineer seeing this architecture for the first time. Here is everything we've planned. What's missing? What would break? What haven't we considered?"

1. Git repo lives WHERE? Absolute paths required. Plan said "project doc directory is a git repo" without specifying path resolution. Fix: git repo root stored in SQLite project record; all paths resolved to absolute before git operations.
2. Frontend holds stale file content during direct edits Debounce save sends frontend's in-memory copy. If AI accepted an edit to the same file while rep had it open, rep's save overwrites it. Fix: backend must re-read and merge on save, or frontend re-fetches before saving.
3. Claude could hallucinate file paths (path traversal risk) Claude returns file_path in proposed edits JSON. "../../server.js" would write to server code. Fix: validate all paths are within the project directory before any write.
4. Server restart with pending edits — orphaned rows If proposed-edits.json is cleaned up early, pending SQLite rows become orphaned. Fix: never delete proposed-edits.json until all rows for that call are resolved.
5. Git commit message collision If rep accepts edits days after the call, commit message "Mar 25 call" appears at the wrong date in git log. Fix: commit label comes from call record in SQLite (call.label), not generated at accept-time.
6. Version history diff direction ambiguity Clicking a version must show git diff <parent-hash> <this-hash>, not git diff <hash> HEAD. API must return the correct hash pair. Document this contract explicitly.
7. No authentication — document as accepted risk Single-user local tool in Phase 1, no auth needed. Must be explicitly documented so it isn't accidentally deployed to a shared server without adding auth first.
8. Monolithic script block will become unnavigable As fetch() calls grow, the mockup's single script block becomes unmanageable. Convention: one <script> section per feature (calls, files, edits, chat) from the start, without introducing a bundler.
All 8 findings are addressed in the architecture doc (epoch-1-happy-path.md). Items 1, 2, and 3 are the most critical and have specific backend fixes. Items 4–8 are implementation conventions documented in the epoch.
